INSTRUCTIONS FOR REPLIT AGENT: IN, DIRECT GOOGLE OAUTH LOGIN
NO CLERK. NO REPLIT AUTH. NO THIRD-PARTY AUTH SERVICE OF ANY KIND.

The owner has ONE shared Google OAuth client. Its credentials are stored in
his Replit Account Vault as GOOGLE_OAUTH_CLIENT_ID and
GOOGLE_OAUTH_CLIENT_SECRET and are REUSED across all of his apps. Your job
is to wire THIS app to that client. You never generate credentials and you
never send the owner to create new ones.

=====================================================
THE SIX NON-NEGOTIABLE RULES
=====================================================

RULE 1. USE THE SHARED GOOGLE OAUTH CREDENTIALS.
Request GOOGLE_OAUTH_CLIENT_ID and GOOGLE_OAUTH_CLIENT_SECRET via the
secrets panel. The owner's Account Vault will auto-fill them - that is
correct and intended. Do NOT ask him to create new credentials, do NOT
send him to any dashboard to make keys. The same client is shared by all
his apps on purpose.

RULE 2. NO AUTH SERVICE MIDDLEMEN.
Do not install or suggest Clerk, Auth0, Firebase Auth, Supabase Auth,
NextAuth hosted, or Replit Auth. Implement Google OAuth 2.0 / OpenID
Connect directly (passport-google-oauth20, openid-client, or hand-rolled
authorization-code flow) with server-side sessions (cookie sessions backed
by the app database or a session store).

RULE 3. GOOGLE IS THE ONLY LOGIN. NO OTHER LOGIN IS PERMITTED.
No email/password, no phone, no username. The sign-in surface shows exactly
one thing: a "Sign in with Google" button.

RULE 4. GOOGLE LOGIN WORKS FOR THE GENERAL PUBLIC. PERIOD.
Any person with any Google account can sign in on the published site. No
invitations, no allowlists for regular users, no Replit account, no
waitlist. (If the Google Cloud consent screen is in "Testing" mode it only
allows listed test users - tell the owner to set it to "In production" in
Google Cloud Console -> APIs & Services -> OAuth consent screen if public
sign-in is blocked.)

RULE 5. LOGIN IS TWO CLICKS AT MOST, THEN THE USER IS IN THE APP.
Click 1: the "Sign in with Google" button on the landing page - it
navigates DIRECTLY to /api/auth/google, which 302-redirects to Google's
OAuth URL. No intermediate sign-in page.
Click 2: the user picks their Google account on Google's chooser.
After the callback they land INSIDE the app. No extra confirmations, no
"continue" screens. Five-click flows are a failure.

RULE 6. ADMIN TAB FOR THE OWNER ONLY.
When the signed-in user's email is johnmichaelkuczynski@gmail.com, the app
shows an extra tab named "Administrative". It contains login data: total
users, a list of users with last sign-in time, and GRAPHS (sign-ins per
day, new users per day). No other user may see this tab or fetch its data.
Enforce on the SERVER (admin middleware that checks the session's email),
not just by hiding the tab in the UI. Record a login event server-side
(userId, email, timestamp) on every successful OAuth callback; store the
events in the app's database; chart them.

=====================================================
ARCHITECTURE (MANDATORY)
=====================================================

- Server routes:
    GET  /api/auth/google           -> 302 to Google's auth URL
                                       (scope: openid email profile,
                                        prompt=select_account,
                                        state parameter for CSRF)
    GET  /api/auth/google/callback  -> exchange code for tokens, verify the
                                       ID token, upsert the user (googleId,
                                       email, name, avatar) in the app DB,
                                       create the session, INSERT a login
                                       event row, then redirect to "/".
    POST /api/auth/logout           -> destroy session, redirect to "/".
    GET  /api/auth/me               -> current user JSON or 401.
- Sessions: httpOnly, secure, sameSite=lax cookies; session store in the
  app's database (e.g. connect-pg-simple) or signed cookie sessions.
  SESSION_SECRET from secrets. No JWTs in localStorage, no Bearer headers.
- The landing page "Sign in with Google" button is a plain link/form to
  /api/auth/google (Rule 5) - never a client-side popup library.
- The root path "/" stays public: signed-out visitors see a landing page
  with ONE "Sign in with Google" button; all other routes are gated:
  client redirects signed-out users to "/", server returns 401 from every
  protected /api route without a session.
- CORS: NEVER cors({ credentials: true, origin: true }). Allowlist only
  the app's own domains (REPLIT_DOMAINS, REPLIT_DEV_DOMAIN, localhost).
- Admin endpoints (Rule 6): requireAdmin middleware on top of requireAuth
  rejects with 403 unless the session email is
  johnmichaelkuczynski@gmail.com. All /api/admin/* routes (user list,
  login-event aggregates for the graphs) use it.
