INSTRUCTIONS FOR REPLIT AGENT: GOOGLE-ONLY LOGIN (CLERK)
NEW PER-APP CREDENTIALS. PRODUCTION KEYS ON THE LIVE SITE.

=====================================================
THE SIX NON-NEGOTIABLE RULES
=====================================================

RULE 1. GENERATE AND USE NEW CLERK CREDENTIALS FOR THIS APP.
Never reuse keys from any other app. If any Clerk keys already exist in this
app's Secrets, assume they were copied from another app: DELETE them and
generate new ones for THIS app. One app = its own credentials. No exceptions.

RULE 2. THE LIVE (PUBLISHED) SITE MUST RUN ON REAL CLERK PRODUCTION
CREDENTIALS, NOT TEST CREDENTIALS.
pk_test_ / sk_test_ keys are for the development preview only. The published
site must run on pk_live_ / sk_live_ production credentials with no limits
and no "Development mode" badge. Do not declare this task finished until the
production-credential step (below) is done or explicitly scheduled with the
user for publish time.

RULE 3. GOOGLE IS THE ONLY LOGIN. NO OTHER LOGIN IS PERMITTED.
No email/password, no phone, no other provider. The sign-in card shows
exactly one thing: a "Continue with Google" button. If an email field
appears, the provider configuration is wrong - fix the configuration, never
hide fields with CSS.

RULE 4. GOOGLE LOGIN WORKS FOR THE GENERAL PUBLIC. PERIOD.
Any person with any Google account can sign in on the published site. No
invitations, no allowlists for regular users, no Replit account, no waitlist
mode. If Clerk's "Restricted" sign-up mode is on, turn it off.

RULE 5. LOGIN IS TWO CLICKS AT MOST, THEN THE USER IS IN THE APP.
Click 1: the "Sign in with Google" button on the landing page. This button
must launch Google OAuth DIRECTLY - use Clerk's
authenticateWithRedirect({ strategy: "oauth_google", ... }) on the button,
NOT a navigation to a separate sign-in page first.
Click 2: the user picks their Google account on Google's chooser.
After that they land INSIDE the app. No intermediate pages, no extra
confirmations, no "continue" screens. Five-click flows are a failure.

RULE 6. ADMIN TAB FOR THE OWNER ONLY.
When the signed-in user's email is johnmichaelkuczynski@gmail.com, the app
shows an extra tab named "Administrative". It contains login data: total
users, a list of users with last sign-in time, and GRAPHS (sign-ins per day,
new users per day). No other user may see this tab or fetch its data.
Enforce on the SERVER (admin middleware that checks the Clerk session's
email), not just by hiding the tab in the UI. Record a login event
server-side (userId, email, timestamp) when a session starts; store events
in the app's database; chart them.

=====================================================
ARCHITECTURE (MANDATORY)
=====================================================

- Two-click flow (Rule 5): the landing page's "Sign in with Google" button
  calls Clerk's authenticateWithRedirect({ strategy: "oauth_google",
  redirectUrl: "/sign-in/sso-callback", redirectUrlComplete: "/" }) via
  useSignIn().signIn - it does NOT navigate to a sign-in page. Mount
  AuthenticateWithRedirectCallback (or the SignIn component) at /sign-in/*
  so the OAuth callback resolves.
- Keep /sign-in and /sign-up routes as fallbacks with routing="path" and
  explicit path props. With Google-only configuration both show just the
  Google button.
- ClerkProvider gets signInUrl="/sign-in" and signUpUrl="/sign-up".
- NEVER link or redirect to any accounts.*.dev URL and never use
  RedirectToSignIn to the hosted portal. If the browser ever leaves the
  app's own domain during login (other than Google's consent screen), the
  wiring is wrong - stop and fix.
- (Path A only) Frontend key via publishableKeyFromHost(
  window.location.hostname, import.meta.env.VITE_CLERK_PUBLISHABLE_KEY)
  from @clerk/react/internal plus unconditional proxyUrl, exactly as the
  skill shows. (Path B: plain publishableKey prop.)
- Sign-in/sign-up routes must tolerate sub-paths (/sign-in/*) so Google's
  OAuth callback (/sign-in/sso-callback) resolves.
- The root path "/" stays public: signed-out visitors see a landing page
  with ONE "Sign in with Google" button; all other routes are gated and
  redirect signed-out users to "/".

=====================================================
SERVER
=====================================================

- (Path A only) Clerk proxy middleware before any body parsers.
- clerkMiddleware() before API routes; every protected route gated by
  requireAuth checking getAuth(req); return 401 without a session.
- CORS: NEVER cors({ credentials: true, origin: true }). Allowlist only the
  app's own domains (REPLIT_DOMAINS, REPLIT_DEV_DOMAIN, localhost).
- Cookie-based auth: no Authorization Bearer headers, no getToken() in the
  web client.
- Admin endpoints (Rule 6): a requireAdmin middleware on top of requireAuth
  that loads the Clerk user for the session and rejects with 403 unless
  their primary email is johnmichaelkuczynski@gmail.com. All /api/admin/*
  routes (user list, login-event aggregates for the graphs) use it.
